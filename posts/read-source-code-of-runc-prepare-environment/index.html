本篇文章介绍如何准备 runC 源码分析的环境，以及 Golang 项目常见的项目结构。runC 的版本为 v1.1.2。

编译带 debug 信息的 runC

编译过程比较简单，先给出所有的命令：

$ git clone https://github.com/opencontainers/runc.git && cd runc
$ git checkout v1.1.2
$ export GCFLAGS="all=-N -l"
$ sed -ri 's/^(GO_BUILD := .+) -trimpath (.+)$/\1 \2/' Makefile
$ make BUILDTAGS="seccomp" runc

对上面的命令做几点说明：

  • 通过 GCFLAGS 环境变量给 Go 编译器传递编译选项，告诉 Go 编译器对项目中所有的包不要进行优化，不要生成内联函数。对于编译选项的作用，可以参考官方文档1。
  • sed 命令的目的是移除 Makefile 中 GO_BUILD 变量中的 -trimpath 字符串。-trimpath 选项的作用是移除源代码文件路径的前缀。注意这个选项是 go build 命令的选项。关于这个选项的详细解释可以参考 stackoverflow 上的这个回答2。
  • 给 make 命令传递 BUILDTAGS 环境变量的目的是启用 runC 对 seccomp 的支持。

Go 项目常用的项目结构

Go 项目常用的项目结构有两种：

  • 一种是像 runC 这样，所有的 main 包的源代码和子包都放在仓库根目录下。这种结构的缺点是源代码与其它脚本和配置文件混在一起，根目录比较混乱。
  • 另一种结构可以参考开源项目 Helm3。这种结构用得更广泛，main 包的代码放在 cmd 目录中，所有的子包放在 pkg 目录中，internal 包中的函数和结构体等不会对外导出，只能在项目内部使用；与测试相关的文件放在 testdata 目录中，scripts 目录中保存项目中用到的脚本。

如何阅读 runC 源码

runC 是一个命令行程序，使用 urfave/cli 库解析命令行参数。urfave/cli 库使用 cli.App 结构体描述一个命令行程序。需要注意 cli.App.Before 成员，库在解析命令行参数前会调用此成员指定的函数。

每个子命令用 cli.Command 结构体表示。所有的子命令结构体保存在 app.Commands 变量中。每个子命令中，Flags 成员保存了所有的选项参数，Action 成员是子命令的入口函数。

例如我们要了解 spec 命令的运行过程，在 main.go 中的 app.Commands 中可以看到，spec 命令对应的 cli.Command 实例为 specCommand。接着查看 specCommand 实例的 Action 成员即可定位到 spec 命令的入口。
